## Part 2：AI 配置加"测试连接"按钮

### 文件

- `app/src/main/java/com/example/todoalarm/ui/SettingsPanel.kt` — 找到 `PlanningAiProviderDialog`（编辑单个 AI 源的对话框）
- `app/src/main/java/com/example/todoalarm/data/PlanningAiCaller.kt` — 复用现有 `callSingle` 或新增轻量测活方法

### 需求

用户的 baseUrl 或 apiKey 可能过期或填错。在 `PlanningAiProviderDialog` 中（保存按钮旁边）加一个"测试连接"按钮：

- 点击后用当前对话框里正在填写的 name/baseUrl/apiKey/model 发起一次最小请求
- 请求内容：`system: "你是一个测试助手，只回 ok"`、`user: "ping"`，期待返回任意非空 content
- 超时 10 秒（比正式调用的 45s 更短）
- 测试期间按钮文字变成"测试中..."并禁用
- 结果显示在按钮下方：
  - 成功：绿色"✓ 连接成功，模型：xxx，响应：xxx 字"
  - 失败：红色"✗ 失败：HTTP 401" 或具体错误
- 测试结果只显示 5 秒后自动消失（避免污染 UI），或用户改字段时清掉

### 实现建议

在 `PlanningAiCaller` 新增：

```kotlin
suspend fun testProvider(provider: PlanningAiProvider): TestResult {
    return try {
        val response = callSingleWithCustomTimeout(
            provider = provider,
            request = PlanningAiRequest(systemPrompt = "你是测试助手", prompt = "ping"),
            connectTimeoutMs = 5_000,
            readTimeoutMs = 10_000
        )
        TestResult.Success(contentLength = response.content.length)
    } catch (e: PlanningAiHttpException) {
        TestResult.Failure("HTTP ${e.statusCode}: ${e.message?.take(120)}")
    } catch (e: Exception) {
        TestResult.Failure(e.message?.take(120) ?: "未知错误")
    }
}

sealed class TestResult {
    data class Success(val contentLength: Int) : TestResult()
    data class Failure(val message: String) : TestResult()
}
```

注意：测活不应该污染正式的多源轮询逻辑，测活就是单次调用单个 provider，不走 fallback。

## Part 4：公告功能

### 用户场景

用户希望能在每日看板顶部显示一条公告横幅，例如：

- 设置公告文本：`"5.16-7.1 期间禁止游玩舞萌DX游戏"`
- 设置生效期：开始日期 `2026-05-16` 到结束日期 `2026-07-01`
- 在生效期内，每日看板顶部出现橙色横幅 + 喇叭图标 + 文字
- 文字超出可显示宽度时左右滚动（marquee 效果）
- 过期后横幅自动消失

点击进入对话框 `SettingsSection.ANNOUNCEMENT`：

### 数据获取

`TodoWidgetService` 的 `RemoteViewsFactory` 需要从 Room 读取今日待办：

```kotlin
class TodoWidgetFactory(private val context: Context) : RemoteViewsService.RemoteViewsFactory {
    private var todos: List<TodoItem> = emptyList()

    override fun onDataSetChanged() {
        val app = context.applicationContext as TodoApplication
        runBlocking {
            val today = LocalDate.now()
            todos = app.repository.getActiveTodos()
                .filter { it.hasDueDate && /* 今天 */ }
                .sortedBy { it.dueAtMillis }
                .take(5)
        }
    }
    // ...
}
```

### 数据更新触发

每次完成/创建/删除待办时通知小组件刷新：

```kotlin
fun notifyWidgetDataChanged(context: Context) {
    val intent = Intent(context, TodoWidgetProvider::class.java).apply {
        action = AppWidgetManager.ACTION_APPWIDGET_UPDATE
        val ids = AppWidgetManager.getInstance(context).getAppWidgetIds(
            ComponentName(context, TodoWidgetProvider::class.java)
        )
        putExtra(AppWidgetManager.EXTRA_APPWIDGET_IDS, ids)
    }
    context.sendBroadcast(intent)
}
```

在 `TodoRepository` 的 `addTodo` / `completeTodo` / `deleteTodo` 等方法成功后调用 `notifyWidgetDataChanged`。

### 主题适配

- `widget_todo.xml` 用 `@android:color/system_neutral1_50`（亮色）/ `@android:color/system_neutral1_900`（深色）作为背景
- 文字色用 `@android:color/system_neutral1_900` / `@android:color/system_neutral1_50`
- 这样跟随系统主题；不强制跟应用内主题模式

## Part 6：缩短启动动画

### 当前状态

`app/src/main/java/com/example/todoalarm/ui/DashboardScreen.kt` 第 177 行：

```kotlin
LaunchedEffect(Unit) {
    delay(1600)
    launchVisible = false
}
```

启动屏 `LaunchScreen()` 显示 1.6 秒。

### 修改

把 `delay(1600)` 改为 `delay(600)`。

理由：
- 1.6 秒在国内 App 标准里偏长
- 600ms 足够让 cold start 时数据库 / 设置加载完成（Compose 第一帧通常已经在 200-400ms 出现）
- 不彻底去掉启动屏，因为它有遮挡 cold start 黑屏的作用

如果 GPT 测试发现 600ms 仍然有数据闪烁（比如 LazyColumn 还没渲染就已经隐藏启动屏），可以加到 800ms。但不要超过 1000ms。
