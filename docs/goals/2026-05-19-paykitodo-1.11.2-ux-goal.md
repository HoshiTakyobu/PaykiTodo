# 第一部分：规划台导入 preview 批量操作

## 问题

规划台识别 → preview 后，每条候选都要单独点开设置"加入倒数日"、"创建链接待办"、"启用打卡"。期末一次导入 30 条 DDL 时极其痛苦。

## U1. preview 顶部加全局操作栏

### 文件
`app/src/main/java/com/example/todoalarm/ui/PlanningDeskPanel.kt` — 规划台 import preview sheet 区域

### 改动

在 preview sheet 的"全选/反选"行下方新增一行全局操作 chip：

1. **「全部加入倒数日」** — Switch，开启后所有选中的 TODO 类候选的 `countdownEnabled` 设为 true
2. **「全部创建待办」** — Switch，开启后所有选中的 EVENT 类候选的 `createLinkedTodo` 设为 true
3. **「全部启用打卡」** — Switch，开启后所有选中的 EVENT 类候选的 `checkInEnabled` 设为 true
4. **「统一分组」** — 点击弹出分组选择 chip 组，选中后所有选中候选的 `groupName` 统一设置

### 交互细节
- 全局 Switch 只影响**当前选中**的候选（未勾选的不动）
- 用户仍可逐条覆盖（单条设置优先级高于全局）
- 全局操作栏默认折叠为一行"批量设置 ▸"，点击展开，避免 preview 首屏太挤

- [ ] 单条手动关闭倒数日 → 导入后该条不带 countdownEnabled，其余仍带

# 第二部分：AI 识图直接进入 preview

## 问题

当前流程：选图 → 等 AI 10-30s → Markdown 追加到文档末尾 → 用户再点"识别"按钮 → 又等一轮 parse → 才进 preview。两次等待 + 一次手动操作。

## U2. 识图完成后自动进入 preview

### 文件
`app/src/main/java/com/example/todoalarm/ui/PlanningDeskPanel.kt`（约 `:343` 附近的 vision 回调）

### 改动

AI vision 返回 Markdown 后：
1. **仍然追加到文档末尾**（保留现有行为，作为备份和可编辑源）
2. **同时**把返回的 Markdown 直接喂给 `PlanningMarkdownParser` + `PlanningRecognitionService`（local parse）
3. 如果 parse 出 ≥1 条候选 → **自动弹出 preview sheet**，不需要用户再点"识别"
4. preview 顶部固定一个**原图缩略图**（小尺寸，可点击放大），方便对照核对
5. preview 底部加一个"返回编辑器修改"文字按钮 → 关闭 preview，光标跳到追加位置

### 如果 parse 出 0 条候选
- 仍追加 Markdown 到文档
- toast "识别结果已追加到文档末尾，请检查格式后手动点「识别」"
- 不弹 preview

- [ ] 选一张课表截图 → AI 返回 5 行 → 自动弹 preview 含 5 条候选
- [ ] preview 顶部能看到原图缩略图

# 第五部分：规划台「今日笔记」快捷入口

## 问题

用户每天新建一个文档写今天的事，但不会写日期 heading。无日期项在 parser 里缺少日期上下文。

## U5. 「今日笔记」入口 + 文档日期上下文

### 文件
- `app/src/main/java/com/example/todoalarm/ui/PlanningDeskPanel.kt`
- `app/src/main/java/com/example/todoalarm/data/PlanningNote.kt`
- `app/src/main/java/com/example/todoalarm/data/PlanningMarkdownParser.kt`

### 改动

#### 5a. PlanningNote 新增字段
```kotlin
val documentDateEpochDay: Long? = null
```
数据库 `MIGRATION_18_19`：`ALTER TABLE planning_notes ADD COLUMN documentDateEpochDay INTEGER DEFAULT NULL`

#### 5b. 规划台顶部加「今日」按钮
在文档选择器旁边（或工具栏左侧）加一个小型 pill 按钮 `今日`：
- 点击 → 查找标题匹配今天日期（`M月d日` / `yyyy-MM-dd` / `M.d`）的非归档文档
- 找到 → 直接打开
- 没找到 → 创建新文档，标题 = 今天日期（如 "5月19日"），`documentDateEpochDay` = today
- 创建时初始内容为空（占位提示已有）

- 这样用户在"今日笔记"里写无日期日程，parse 后候选的 startAt/endAt 自动落在今天

#### 5d. 桌面端同步
`/api/planning/notes` 响应和 `/api/planning/save` 请求中加入 `documentDateEpochDay` 字段。

# 第六部分：打卡忘记签退的自动兜底

## 问题

学生离开图书馆忘了点签退，第二天发现"今日投入 14 小时"，AI 日报数据被严重污染。当前的"日程结束时自动签退"只在用户主动点"完成日程"时触发——如果用户没点完成，系统不会兜底。

## U6. 闲置自动签退 watchdog

### 文件
- 新建 `app/src/main/java/com/example/todoalarm/alarm/EventCheckInWatchdog.kt`
- `app/src/main/java/com/example/todoalarm/data/AppSettingsStore.kt`
- `app/src/main/java/com/example/todoalarm/TodoApplication.kt`
- `app/src/main/java/com/example/todoalarm/widget/TodoWidgetService.kt`（顺便兜底）

### 改动

#### 6a. 设置项
设置 → 日历与提醒 → 已有打卡设置区下方新增：
- "闲置自动签退阈值" 下拉：`关闭 / 1 小时 / 2 小时（默认）/ 4 小时 / 8 小时`
- 字段名 `eventCheckInIdleAutoCheckOutHours: Int`，默认 2

#### 6b. Watchdog 逻辑
`EventCheckInWatchdog.runOnce(context)`：
1. 读取设置阈值（关闭则直接返回）
2. 查询所有 `event_check_ins` 中 `checkOutAtMillis IS NULL` 的活跃记录
3. 对每条活跃记录，找到对应 event：
   - 如果 event 已删除（孤儿记录）→ 以 `checkInAtMillis + 阈值` 签退
   - 如果 event 的 `endAtMillis` 已过去 ≥ 阈值小时 → **以 event 的 `endAtMillis` 签退**（不是当前时间，因为这样更接近真实情况）
   - 重算 `totalCheckInMinutes` 写回 todo_items
4. 每条自动签退发一条低优 notification（channel `event_checkin_auto_checkout`）：
   - 标题："已自动签退"
   - 内容：`"{日程标题}" 在 {endAt 时间} 自动签退，本次投入 Xm`
   - 点击 → 打开该日程详情

#### 6c. 触发时机
- App `onCreate` 启动后 5 秒延迟触发一次（避免阻塞冷启动）
- App resume 时触发
- TodoWidgetService 每分钟 tick 时触发（widget 已有刷新闭环，复用即可）

#### 6d. 备份兼容
`AppSettings` 序列化加 `eventCheckInIdleAutoCheckOutHours` 字段。

- [ ] 设置项显示并可切换阈值
- [ ] 模拟一条 event：endAt 在 3 小时前，活跃签到 → app resume → 自动签退 + 通知
- [ ] 自动签退时间是 endAt，不是 now
- [ ] 阈值设为"关闭" → 不触发

# 第七部分：桌面同步保活 WiFi

## 问题

桌面端 Web 在期末写论文时是高频场景，但手机熄屏后 socket 立刻断。用户要么手动设置常亮、要么频繁戳屏幕。

## U7. DesktopSyncService 持有 WiFi 锁 + Wake 锁

### 文件
- `app/src/main/java/com/example/todoalarm/sync/DesktopSyncService.kt`
- `app/src/main/AndroidManifest.xml`（确认 WAKE_LOCK 权限已声明）
- `app/src/main/java/com/example/todoalarm/data/AppSettingsStore.kt`

### 改动

#### 7a. 设置项
设置 → 桌面同步 区域加 Switch：
- "桌面同步期间保持网络唤醒" — 默认 **开启**
- 副标题文案："保证浏览器在你熄屏时仍能连接，会增加少量电量消耗。"
- 字段名 `desktopSyncWifiKeepAlive: Boolean`

#### 7b. Service 中持锁
`DesktopSyncService.kt:25` 类内部新增：
```kotlin
private var wifiLock: WifiManager.WifiLock? = null
private var wakeLock: PowerManager.WakeLock? = null
```

`onStartCommand` 启动 server 后，如果设置为开：
```kotlin
val wm = getSystemService(WIFI_SERVICE) as WifiManager
wifiLock = wm.createWifiLock(
    WifiManager.WIFI_MODE_FULL_HIGH_PERF,
    "PaykiTodo:DesktopSync"
).apply { acquire() }

val pm = getSystemService(POWER_SERVICE) as PowerManager
wakeLock = pm.newWakeLock(
    PowerManager.PARTIAL_WAKE_LOCK,
    "PaykiTodo:DesktopSync"
).apply { acquire() }
```

`onDestroy` / `stopSelf` 前释放（`if (wakeLock?.isHeld == true) wakeLock?.release()`，wifiLock 同理），并把 holder 置 null。

#### 7c. 前台通知文案
通知 builder 内容文字改为：`"桌面端连接中 · 已保持网络唤醒"`（开启时） / `"桌面端连接中"`（关闭时）。

#### 7d. AndroidManifest 检查
确认已有 `<uses-permission android:name="android.permission.WAKE_LOCK"/>`。如果没有就加上。

# 实现顺序建议

按"风险从低到高、用户感知从强到弱"排序。前几轮做完用户立刻能感受到改善，后几轮处理底层兜底。

1. **第一轮 U9（看板卡片折叠）** — 纯 UI，零数据风险，立竿见影
2. **第二轮 U8（抽屉精简）** — 纯 UI，无数据迁移
3. **第三轮 U4（交集/并集切换）** — 仅 ViewModel + UI，无 schema 变更
4. **第四轮 U1（preview 批量操作）** — 纯 preview UI，不涉及 DAO
5. **第五轮 U2（识图直进 preview）** — 复用现有 parser + preview，胶水代码
6. **第六轮 U5（今日笔记 + documentDateEpochDay）** — 含数据库 18→19 迁移，单独一轮
7. **第七轮 U6（自动签退 watchdog）** — 新逻辑 + 通知 channel
8. **第八轮 U7（WiFi 保活）** — 涉及前台 service 生命周期，最后做避免影响其他验证
9. **第九轮 U3（看板快速签到）** — 跨 UI + 数据写入 + 设置持久化，工作量最大放最后

# 致执行端的提醒

1. **U2（识图直进 preview）和 U5（今日笔记）有 parse 路径耦合**：U5 引入了 `documentDateEpochDay` 上下文，U2 把识图 Markdown 喂给 parser 时要传入正确的 noteId 才能继承日期上下文。建议先做 U5 再做 U2，或两轮合并。

2. **U3（快速签到）和 U6（自动签退）也有耦合**：快速签到创建的日程默认时长 30min，如果用户超时未签退，U6 watchdog 会在阈值后自动签退。务必确认 U6 对快速签到日程的处理符合预期（建议视为普通 event 处理即可）。

3. **U7（WiFi 锁）涉及电量**：实测确认通知文案诚实告知用户。不要默认隐藏开关，给用户选择权。

4. 所有持久化字段加入 `AppSettings` 时，**不要忘记在 `BackupManager` 的 `AppSettings.toJson()` / `fromJson()` 加序列化**。这是 1.11.0 时已经踩过的坑。
