## 0. 第一轮用户确认结论

本节记录 2026-05-14 用户对设计草案的确认和补充，后续实现应以本节为准。

### 0.1 已确认规则

1. 第一阶段需要支持多篇规划文档，而不是只做一篇默认文档。
   - 打开规划台时默认打开最后一次打开的文档。
   - 支持像备忘录一样新建文档、打开历史文档。
2. `- [ ] 10:00-11:30 写论文` 这类轻量日程行，默认优先识别为日程。
   - 同时以该日程的终止时间创建一条待办。
   - 该待办的 DDL 等于日程结束时间。
3. 未写 `#remind` 时，默认提前 5 分钟全屏提醒 + 响铃 + 震动。
4. `#ddl 5.28` 这类只有日期、没有时间的 DDL，默认当天 23:59。
5. 第一阶段不做完整项目树。
   - 大任务下面的小任务先转为独立待办。
   - 子任务备注中记录所属大任务。
6. 电脑端第一阶段接受 `textarea + 右侧识别预览` 的简单实现。

### 0.2 用户新增要求

1. 任务拆解和日期识别第一阶段应以本地规则解析为主，不能依赖付费模型。
2. DDL 和时间解析要覆盖用户可能自然敲出的常见格式。
3. 标签是高级玩法，不能成为日常输入负担。
4. 手机端需要输入辅助栏，解决手打 `- [ ]`、缩进、标签很麻烦的问题。
5. 手机端要支持大任务下面继续输入小任务，不能强迫用户理解 Markdown 缩进。

### 0.3 本轮后仍需确认

1. 轻量日程行“同时创建待办”是否默认勾选但允许在预览中取消？
2. 如果行首只有 `10:00-12:30 作业1`，且没有日期上下文，是默认今天，还是在识别预览中要求确认日期？
3. 手机端快捷栏是只在输入法弹出时显示，还是常驻编辑器底部？
4. 多文档第一阶段是否需要搜索文档内容，还是只做文档标题列表？
5. 文档中的已导入条目是否需要自动追加 `#imported` 或其他标记，避免重复导入？

## 1. 背景

PaykiTodo 当前已经具备较强的待办、日程、提醒、全屏提醒和电脑同步能力，但这些能力主要覆盖任务管理链路的最后一环。

用户当前真实工作流是：

```text
先在纸上 / Word / Excel / 备忘录中写出接下来要做什么
  -> 再大致拆分任务
  -> 再粗略规划每一天
  -> 再对明天或今天做细化时间安排
  -> 最后才录入 PaykiTodo 的待办 / 日程 / DDL
```

因此，PaykiTodo 如果只优化最后的提醒和日程编辑，会仍然错过用户实际使用链路的前半段。

规划台要解决的问题是：

```text
让 PaykiTodo 从“最终提醒工具”前移为“从想法到执行计划”的入口。
```

## 2. 目标

### 2.1 第一阶段目标

第一阶段不做复杂拖拽、不做甘特图、不做过重的项目管理系统。

先做一个类似备忘录 / Obsidian 的自由文本规划界面：

- 能像备忘录一样自由写 Markdown。
- 支持多篇规划文档，默认打开最后一次打开的文档。
- 能识别 `- [ ]` 待办项。
- 能识别大任务下面的小任务。
- 能识别简单标签，例如 `#task`、`#ddl 5.28`、`#today`、`#schedule 10:00-11:30`。
- 能把识别出的任务转为 PaykiTodo 待办。
- 能把识别出的时间段转为 PaykiTodo 日程。
- 手机端和电脑端都能使用。

### 2.2 非目标

第一阶段暂不做：

- 拖拽任务到某一天。
- 拖拽任务到日历时间轴。
- 甘特图。
- 复杂项目依赖关系。
- 多人协作。
- 云同步。
- AI 自动规划。

这些可以作为后续功能，但不是第一版的重点。

## 3. “规划台”和“收集箱”的关系

用户提出：没有完全弄明白规划台和收集箱的区别。

建议定义如下：

### 3.1 规划台

规划台是一个大的工作区。

它像一个可编辑的 Markdown 文档，用于写：

- 接下来要做什么；
- 大任务和小任务；
- 周计划；
- 日计划；
- 临时想法；
- 可以转成待办或日程的条目。

规划台是“写、想、拆、整理”的地方。

### 3.2 收集箱

收集箱是规划台里的一个特殊区域或默认文档。

它用于快速丢进去还没整理的内容。

例如：

```markdown
# 收集箱

- [ ] 整理保研材料
- [ ] 写课程论文
- [ ] 复习音乐史
- [ ] PaykiTodo 规划台想法
```

收集箱是“先别管结构，先收进去”的地方。

### 3.3 第一阶段实现建议

第一阶段可以不强行把两者分成两个复杂模块。

可以在抽屉中新增一个入口：

```text
规划台
```

进入后默认打开一个 Markdown 文档，文档顶部可以有几个快捷分区：

```markdown
# 收集箱

# 本周计划

# 今日三件大事

# 任务拆解
```

这样用户不会被复杂模块吓退。

## 4. Markdown 任务语法设计

设计原则：

- 尽量接近日常书写习惯。
- 不强迫用户一开始写完整语法。
- `- [ ]` 是核心识别标志。
- 标签是增强信息，不是必填。
- 没写标签也可以作为普通待办候选。

## 5. 基础任务写法

### 5.1 普通任务

```markdown
- [ ] 整理保研材料
- [ ] 写课程论文
- [ ] 复习中国音乐史
```

识别结果：

- 每一行都是一个可转化任务。
- 未写 DDL 时，默认转成无 DDL 待办，或让用户在预览界面补 DDL。

### 5.2 已完成任务

```markdown
- [x] 整理成绩单
```

识别结果：

- 默认不转入新待办。
- 可以在预览中显示为“已完成，默认跳过”。

### 5.3 大任务和小任务

推荐使用 Markdown 缩进表达父子关系：

```markdown
- [ ] 保研材料准备 #project #ddl 5.28
  - [ ] 整理成绩单 #ddl 5.18
  - [ ] 整理获奖证明 #ddl 5.20
  - [ ] 写个人陈述初稿 #ddl 5.24
  - [ ] 联系导师 #ddl 5.26
```

识别结果：

- 第一行可以识别为项目 / 大任务。
- 缩进子项识别为子任务。
- 如果第一阶段不做复杂项目系统，可以先把子任务转为独立待办，并在备注中写入：

```text
所属大任务：保研材料准备
```

这样既保留层级信息，又不需要一开始重构数据库。

### 5.4 多层级任务

允许写，但第一阶段只识别到两层最稳：

```markdown
- [ ] 课程论文 #project
  - [ ] 确定选题
  - [ ] 查资料
    - [ ] 找 5 篇论文
    - [ ] 整理引用
```

第一阶段处理建议：

- 一级：大任务 / 项目候选。
- 二级：待办候选。
- 三级及以下：作为上级任务备注保留。

后续如果做项目树，再完整支持多级。

## 6. 标签语法设计

### 6.1 基础标签

```markdown
#task
#project
#today
#tomorrow
#important
```

含义：

| 标签 | 含义 |
|---|---|
| `#task` | 明确表示这是一条任务 |
| `#project` | 表示这是一个大任务 / 项目 |
| `#today` | 倾向安排到今天 |
| `#tomorrow` | 倾向安排到明天 |
| `#important` | 标记为重要 |

说明：

- `- [ ]` 本身已经足够识别任务。
- `#task` 只是增强标记，适合像 Obsidian task 插件一样的习惯。

### 6.2 DDL 标签

推荐支持：

```markdown
#ddl 5.28
#ddl 05-28
#ddl 2026-05-28
#ddl 明天
#ddl 今天 16:30
#ddl 5.28 23:59
```

默认规则：

- `5.28`、`05-28`：缺省年份，默认当前年。
- 只有日期没有时间：默认 23:59。
- `今天 16:30`：当天 16:30。
- `明天`：明天 23:59。

示例：

```markdown
- [ ] 交课程论文 #ddl 5.28
- [ ] 今晚整理材料 #ddl 今天 22:30
```

### 6.3 分组 / 目标标签

推荐支持：

```markdown
#group 保研
#group 课程
#goal 保研
#goal PaykiTodo开发
```

第一阶段可以先映射到现有“分组”。

例如：

```markdown
- [ ] 整理导师名单 #group 保研 #ddl 5.20
```

识别结果：

- 标题：整理导师名单
- 分组：保研
- DDL：2026-05-20 23:59

### 6.4 提醒标签

推荐支持：

```markdown
#remind 5
#remind 5,15
#remind 16:30
#remind 05-28 15:00
```

复用当前 PaykiTodo 已有提醒时间输入规则：

```text
5,15,16:30,05-10 15:00,2026-05-10 14:30
```

如果没有写 `#remind`，第一阶段建议默认：

- 待办：如果有 DDL，默认提前 5 分钟全屏提醒，可在设置中调整。
- 日程：默认开始前 5 分钟全屏提醒，可在设置中调整。

这里符合用户希望“简单写，强提醒默认补上”的诉求。

### 6.5 日程标签

推荐支持：

```markdown
#schedule 10:00-11:30
#schedule 2026-05-20 10:00-11:30
#schedule 明天 14:00-16:00
```

示例：

```markdown
- [ ] 复习音乐史 #schedule 明天 14:00-16:00
```

识别结果：

- 转为日程，而不是普通待办。
- 标题：复习音乐史。
- 时间：明天 14:00-16:00。
- 默认提醒：开始前 5 分钟全屏提醒。

## 7. 更轻量的日程自然写法

用户提到点选式添加日程不自然，更习惯写：

```text
几点到几点干什么
```

因此规划台中应额外支持“轻量日程行”。

### 7.1 推荐写法

```markdown
- [ ] 10:00-11:30 复习音乐史
- [ ] 14:00-16:00 写课程论文
- [ ] 明天 19:30-21:00 整理保研材料
- [ ] 5.20 09:00-10:30 联系导师
```

解析规则：

- 行首出现 `HH:mm-HH:mm`：识别为日程候选。
- 行首出现 `明天 HH:mm-HH:mm`：识别为明天日程。
- 行首出现 `M.d HH:mm-HH:mm`：识别为当前年的指定日期日程。
- 标题取时间段后面的文字。
- 默认提醒按设置补上，例如开始前 5 分钟全屏提醒。

### 7.2 为什么需要这个

因为它比完整批量导入语法更接近日常规划：

```markdown
# 明天

- [ ] 09:00-10:30 写论文
- [ ] 10:45-11:30 整理材料
- [ ] 14:00-16:00 复习
```

这符合用户“像在空白文档里写计划”的习惯。

## 8. 文档内日期上下文

为了减少重复写日期，可以支持标题上下文。

例如：

```markdown
# 2026-05-20

- [ ] 09:00-10:30 整理材料
- [ ] 14:00-16:00 写论文
```

或：

```markdown
# 明天

- [ ] 09:00-10:30 复习音乐史
- [ ] 14:00-16:00 写课程论文
```

解析规则：

- 如果任务行没有日期，但上方最近的标题是日期 / 今天 / 明天 / 周几，则继承该日期。
- 如果没有日期上下文，则默认今天，或在预览中要求确认。

## 9. 转化流程设计

规划台不应该用户一写就自动创建待办 / 日程。

建议流程：

```text
编辑 Markdown
  -> 点击“识别”
  -> 进入识别预览
  -> 用户勾选要导入的条目
  -> 用户选择“导入为待办 / 导入为日程 / 自动按识别结果导入”
  -> 写入 PaykiTodo 数据库
```

这样可以避免误识别造成污染。

## 10. 识别预览界面

识别结果建议按类型分组：

```text
待办候选
日程候选
项目 / 大任务候选
无法识别
已完成 / 跳过
```

每条显示：

- 原文；
- 解析出的标题；
- 类型；
- DDL / 起止时间；
- 分组；
- 提醒；
- 是否同时创建待办；
- 是否会导入。

允许用户在预览界面修正。

## 11. 手机端 UI 设计

手机端第一阶段建议做得像备忘录，而不是做复杂项目管理。

入口：抽屉菜单新增 `规划台`。

界面结构：

```text
顶部栏：规划台                  [识别] [更多]

Markdown 编辑区

底部可选：最近识别结果 / 草稿保存状态
```

编辑区要求：

- 支持多行文本。
- 支持 Markdown 原文编辑。
- 不强制所见即所得。
- 第一阶段可以是普通多行输入框。
- 后续再做 Markdown 高亮。

更多菜单：

- 插入今日模板。
- 插入明日模板。
- 插入本周模板。
- 查看语法帮助。

## 12. 电脑端 UI 设计

电脑端很重要，因为用户明确说在电脑上做规划更得心应手。

电脑端规划台应当在现有 Web 控制台中新增一个 Tab：

```text
规划台
```

布局建议：

```text
左侧：Markdown 编辑器
右侧：识别结果预览
顶部：保存草稿 / 识别 / 导入选中
```

第一阶段可以不做复杂编辑器，只用 textarea。

后续可以升级为：

- Markdown 高亮；
- 任务勾选；
- Obsidian 风格双栏预览；
- 快捷键；
- 多文档。

## 13. 数据存储设计草案

第一阶段建议新增一张规划草稿表，而不是把 Markdown 存在设置里。

示例：

```kotlin
PlanningNoteEntity(
    id: Long,
    title: String,
    contentMarkdown: String,
    createdAtMillis: Long,
    updatedAtMillis: Long,
    archived: Boolean
)
```

第一阶段已经确认需要支持多篇规划文档。

需要额外记录最后打开的文档 ID，例如 `lastOpenedPlanningNoteId`，以便下次打开规划台时恢复到上次状态。

## 14. 解析器设计草案

建议新增独立解析器：

```text
PlanningMarkdownParser
```

输入：Markdown 文本。

输出：

```kotlin
PlanningParseResult(
    todos: List<ParsedTodoCandidate>,
    events: List<ParsedEventCandidate>,
    projects: List<ParsedProjectCandidate>,
    skipped: List<ParsedSkippedLine>,
    errors: List<ParsedLineError>
)
```

解析能力第一阶段只覆盖：

- `- [ ]` / `- [x]`。
- 缩进层级。
- `#ddl`。
- `#remind`。
- `#group`。
- `#schedule`。
- 行首轻量日程：`10:00-11:30 标题`。
- 日期上下文标题：`# 明天`、`# 2026-05-20`。

## 15. 默认提醒策略

为了避免语法复杂度劝退，规划台导入时应有默认策略。

建议设置项：

```text
规划台导入待办默认提醒：提前 5 分钟，全屏提醒，响铃 + 震动
规划台导入日程默认提醒：开始前 5 分钟，全屏提醒，响铃 + 震动
```

用户未写 `#remind` 时，自动应用默认值。

用户写了 `#remind` 时，以用户输入为准。

## 17. 后续增强方向

如果第一阶段完成后继续扩展，可以考虑：

1. 周计划视图。

## 17A. 本地解析和自然输入补充规范

本节补充第一轮用户反馈后的解析规则。第一阶段必须本地完成，不依赖 AI 模型。

### 17A.1 本地解析原则

规划台解析器应作为本地规则引擎实现。

优先级建议：

```text
显式标签 #schedule / #ddl / #remind
  > 行首自然日程
  > Markdown checkbox
  > 日期上下文标题
  > 默认导入策略
```

这样可以保证用户写了明确标签时尊重标签；没写标签时，也能识别自然写法。

### 17A.2 需要覆盖的日期格式

| 类型 | 示例 |
|---|---|
| 月日点号 | `5.28`、`05.28` |
| 月日短横线 | `5-28`、`05-28` |
| 中文月日 | `5月28日`、`05月28日` |
| 完整日期短横线 | `2026-05-28` |
| 完整日期点号 | `2026.05.28` |
| 完整中文日期 | `2026年5月28日` |
| 相对日期 | `今天`、`明天`、`后天` |

默认规则：

- 未写年份时，默认当前年。
- 只有日期没有时间时，默认 23:59。
- 如果解析出的 DDL 早于当前时间，识别预览中标红提示，不静默改到明年。

### 17A.3 需要覆盖的时间格式

| 类型 | 示例 |
|---|---|
| 24 小时英文冒号 | `14:30` |
| 24 小时中文冒号 | `14：30` |
| 12 小时制 | `2:30 pm`、`2:30pm`、`11:55PM` |
| 单数字小时 | `9:00`、`2:30 pm` |

### 17A.4 日期和时间组合

日期和时间之间允许空格或逗号：

```text
5.28 14:30
5.28,14:30
5月28日 11:55pm
2026-05-28 14:30
2026年5月28日, 2:30 pm
```

### 17A.5 轻量日程自然写法

第一阶段必须支持用户最自然的写法：

```text
10:00-12:30 作业1
10:00~12:30 作业1
10:00 至 12:30 作业1
10:00到12:30 作业1
明天 19:30-21:00 整理保研材料
5.28 14:30-16:00 写论文
5月28日 11:55pm-次日00:30 整理材料
```

解析规则：

- 行首是时间段时，默认识别为日程。
- 如果行内没有日期，则读取最近的日期上下文标题。
- 如果没有日期上下文，默认策略待进一步确认：默认今天，或在预览中要求确认。
- 标题取时间段后面的文字。
- 导入时默认同时创建一条待办，待办 DDL 等于日程结束时间。
- 识别预览中应允许取消“同时创建待办”。

### 17A.6 日期上下文标题

规划台应支持通过标题提供日期上下文：

```markdown
# 明天

- [ ] 09:00-10:30 写论文
- [ ] 14:00-16:00 复习
```

标题可识别：

```text
# 今天
# 明天
# 后天
# 周五
# 5.28
# 5月28日
# 2026-05-28
```

## 17B. 手机端输入辅助设计

标签和 Markdown 语法属于高级能力，不能让用户在手机中文输入环境里手动找符号。

### 17B.1 输入法上方快捷栏

手机端规划台在输入法弹出时，应显示一条横向快捷栏：

```text
[任务] [子任务] [缩进] [减少缩进] [DDL] [日程] [提醒] [分组] [今天] [明天]
```

行为：

- 点 `任务`：插入 `- [ ] `。
- 点 `子任务`：插入 `  - [ ] `，或把当前行改成子任务。
- 点 `缩进`：当前行增加两个空格缩进。
- 点 `减少缩进`：当前行减少一级缩进。
- 点 `DDL`：插入 `#ddl `。
- 点 `日程`：插入 `#schedule `。
- 点 `提醒`：插入 `#remind `。
- 点 `分组`：插入 `#group `。
- 点 `今天` / `明天`：插入对应日期上下文或标签。
- 长按按钮：显示该按钮用途和例子。

### 17B.2 回车续写规则

为了解决“大任务下面输入小任务”的问题，编辑器应支持列表续写：

- 当前行是 `- [ ] 大任务`，按回车后自动插入下一行 `- [ ] `。
- 点击 `子任务` 后，下一行变成 `  - [ ] `。
- 当前行是 `  - [ ] 子任务`，按回车后继续生成同级子任务 `  - [ ] `。
- 如果当前行是空的子任务行，再按一次回车，则退出缩进，回到 `- [ ] `。

这套行为让用户不必理解 Markdown 缩进，也能自然写出父任务和子任务。

### 17B.3 手机端不强迫完整 Markdown

用户可以直接写：

```text
10:00-12:30 作业1
整理保研材料 5.28
明天 14:00-16:00 复习音乐史
```

识别器应尽量识别这些自然行。`- [ ]` 和标签只是增强准确度，不是唯一入口。
