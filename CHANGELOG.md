# Changelog

PaykiTodo 采用语义化倾向的应用版本号和递增 `versionCode`。本文件面向用户记录重要变化，不再保存每一次内部调试重建的完整流水账；完整历史可通过 Git commit、tag 和 `docs/archive/` 追溯。

## v1.13.28

发布日期：2026-06-01（本地调试构建，尚未创建 GitHub Release）

### 修复

- 手机端待办详情快速预览顶部操作改为直接显示 `取消待办`，不再使用容易误解的 `取消归档` 文案；底部仍保留 `取消待办（归档）`，明确取消会进入历史记录、删除不会进入历史记录。
- 从每日看板、搜索结果、规划台关联项、通知 / 小组件等入口打开未完成待办时，快速预览都保持可取消；只有已完成或已取消的历史待办隐藏取消 / 删除并显示恢复入口。
- 电脑端 Web 待办详情预览顶部新增 `取消待办` 操作，用户不用滚到底部也能从快速预览执行归档取消。

### 构建

- `versionName = 1.13.28`
- `versionCode = 276`
- 数据库版本保持 `27`，不新增数据迁移。

## v1.13.27

发布日期：2026-06-01（本地调试构建，尚未创建 GitHub Release）

### 修复

- 日程全屏提醒页和无障碍覆盖提醒中点击 `我知道了` / `签到` 时，不再清掉进行中日程通知的开始 / 结束闹钟；确认提醒后仍会保留或重排 `进行中日程` 常驻通知，避免日程开始后通知栏没有持续提示。
- 进行中日程通知的通知 ID 改为使用完整事项 ID 哈希，不再只取 `eventId % 10000`，降低大量日程下通知互相覆盖或错误取消的风险。

- `versionCode = 275`

## v1.13.24

发布日期：2026-06-01（本地调试构建，尚未创建 GitHub Release）

### 修复

- 待办详情快速预览现在把取消动作明确标为 `取消待办（归档）`，并在执行前说明取消会进入历史记录，避免用户把关闭、删除和取消混在一起。
- 电脑端 Web 待办预览同样补充取消确认，文案明确取消会归档、删除会直接移除且不进入历史记录。

- 备份 / 恢复会保留循环实例跳过记录，避免恢复备份后已删除的单次循环待办重新出现。

## v1.13.14
