## 用户反馈背景

1.9.1 把 AI 日报 / 周报写入规划台的「AI 日报」/「AI 周报」专属文档。用户反馈这是**不合适的归属**：

- 规划台是用户主动写计划的地方，不应该有自动生成内容混进文档列表
- 报告应该是一个**只读的历史归档**，类似 QQ 动态的「周访客报告」——有独立入口，可以随时翻看过去的报告，但平时不占用主要工作区
- 用户不会去规划台找报告，直觉上会去一个"报告中心"或类似入口

本轮把 AI 报告从规划台彻底搬走，做成独立的归档模块。

### 模型

```kotlin
enum class AiReportType { DAILY, WEEKLY }

@Entity(tableName = "ai_reports")
data class AiReport(
    @PrimaryKey(autoGenerate = true) val id: Long = 0,
    val type: AiReportType,
    val generatedAtMillis: Long,         // 生成时刻
    val periodStartMillis: Long,         // 报告覆盖的时间段开始（日报=当天00:00；周报=本周一00:00）
    val periodEndMillis: Long,           // 报告覆盖的时间段结束
    val content: String,                 // AI 或本地模板生成的正文
    val providerName: String = "",       // 生成时使用的 AI 源名（本地兜底时填"本地模板"）
    val isLocalFallback: Boolean = false
)
```

### DAO

## Part 2：迁移 — 从规划台搬走旧报告

### 关键问题

1.9.1 - 1.9.7 期间已经生成的 AI 日报内容写在规划台的「AI 日报」/「AI 周报」两篇 PlanningNote 里。直接删除这两篇文档会丢失用户的历史报告。

### 迁移策略

`MIGRATION_X_Y` 内：

1. 找到 `planning_notes` 表里 title 等于 `"AI 日报"` 或 `"AI 周报"` 的记录
2. 这两篇文档的内容格式（参考 `DailyReportGenerator.writeToReportNote`）：
   ```
   ## 2026-05-17 周日 22:00

   今天完成 5 条待办...

   ---

   ## 2026-05-16 周六 22:00

   ...

   ---
   ```
   按 `## yyyy-MM-dd` 标题切分成多段
3. 每段作为一条 `AiReport` 插入：
   - `type` = 由文档标题决定（"AI 日报" → DAILY，"AI 周报" → WEEKLY）
   - `generatedAtMillis` = 解析 `## yyyy-MM-dd EEE HH:mm` 标题得出（解析失败则用 PlanningNote 的 `updatedAtMillis` 兜底）
   - `periodStart/EndMillis` = 根据 type 推算：DAILY 用当天 00:00 - 23:59；WEEKLY 用 generatedAt 所在周的周一到周日
   - `content` = 段落正文（去掉标题行）
   - `providerName` = `"历史归档"`
   - `isLocalFallback` = false
4. 迁移完成后，删除这两条 PlanningNote 记录（避免规划台还能看到）
5. 同时清理 `lastOpenedPlanningNoteId` 如果指向被删除的报告文档

由于 Room 迁移用的是 SupportSQLiteDatabase，没有 Kotlin data class 的便利。建议在 SQL 里完成基本的表结构创建，然后通过 `Callback.onOpen` 或单独的一次性数据迁移逻辑（在 `TodoApplication` 启动时检查并执行）来做内容迁移。

**实现建议**：
- `MIGRATION_X_Y` 只做 schema 变更（创建 `ai_reports` 表）
- `TodoApplication.onCreate` 中调用一次性的 `LegacyAiReportMigration.migrateIfNeeded(repository)`，用 SharedPreferences 标记是否已迁移过：
  ```kotlin
  if (!settingsStore.currentSettings().legacyAiReportMigrated) {
      LegacyAiReportMigration.migrate(repository)
      settingsStore.markLegacyAiReportMigrated()
  }
  ```
  这样迁移逻辑可以用 Kotlin 完整能力，不被 Room 迁移 API 限制。

新增 `legacyAiReportMigrated: Boolean = false` 到 `AppSettings`，加 `markLegacyAiReportMigrated()` 方法。

## Part 3：DailyReportGenerator 改写入路径

### 文件

- `app/src/main/java/com/example/todoalarm/data/DailyReportGenerator.kt`

### 改动

1. `writeToReportNote(...)` 改名为 `saveReport(app, type, content, providerName, isLocalFallback)`
2. 内部不再调用 `getOrCreateReportNote / updatePlanningNote`，改为：
   ```kotlin
   private suspend fun saveReport(
       app: TodoApplication,
       type: AiReportType,
       content: String,
       providerName: String,
       isLocalFallback: Boolean,
       periodStartMillis: Long,
       periodEndMillis: Long
   ): Long {
       val report = AiReport(
           type = type,
           generatedAtMillis = System.currentTimeMillis(),
           periodStartMillis = periodStartMillis,
           periodEndMillis = periodEndMillis,
           content = content,
           providerName = providerName,
           isLocalFallback = isLocalFallback
       )
       return app.repository.saveAiReport(report)
   }
   ```
3. `generateDaily` / `generateWeekly` 调用新接口，传入对应的 periodStart / periodEnd
4. 移除 `getOrCreateReportNote` 在 `TodoRepository` 中的实现
5. 移除 `findPlanningNoteByTitle` 在 `TodoDao` 中的实现（如果只是为这个用的）
