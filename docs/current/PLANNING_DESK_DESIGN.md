## 0Q. 1.8.1 AI 触发收紧与测活（2026-05-16 / v1.8.1）

`1.8.1` 不重做 AI 识别架构，只修复用户对“换行就触发 AI”的体感问题并补齐配置验证：

1. 手机端规划台编辑、换行、自动保存、映射同步均不调用 AI；只有顶部 `识别` 按钮会进入 `PlanningRecognitionService.recognize`。
2. 电脑端编辑器 `input` 只清空旧预览、刷新本地预览状态并标记 dirty；只有 `识别` 按钮和 `Ctrl+Enter` 会调用 `/api/planning/parse`。
3. 电脑端导入在没有识别预览时不再隐式调用 `parsePlanningEditor()`，而是提示用户先点击 `识别`，避免“没有主动识别也调 AI”的误解。
4. 手机端自动保存状态不再使用进度条，改为普通 `自动保存中` 文案，避免把本地保存反馈误看成网络/AI loading。
5. 设置页单个 AI 源编辑弹窗增加 `测试连接`，用当前填写的 Base URL / API Key / 模型发起短超时 ping，成功或 HTTP/网络错误会直接显示在弹窗里。
6. `docs/current/AI_RECOGNITION_VERIFICATION.md` 记录了手机端、电脑端调用链和四种 AI 状态的手动验证步骤。

## 0P. 1.8.0 规划台映射闭环与上游同步（2026-05-16 / v1.8.0）

`1.8.0` 的重点不再是“再多识别一种语法”，而是把规划台从一次性导入器推进成真正的上游规划文档：

1. 每次从规划台导入待办或日程后，除了给原文追加 `#imported`，还会在 `planning_line_mappings` 表里记录“规划文档行 <-> 正式事项”的映射。
2. 映射不再只靠固定行号。系统会先用去掉 `#imported` 和空白噪声后的指纹匹配，再在必要时做模糊文本匹配，尽量容忍插行和轻度改写。
3. 映射状态现在分为 `ACTIVE`、`COMPLETED`、`CANCELED`、`ORPHANED`、`CONFLICT`。手机端和电脑端预览都能直接看到这些状态。
4. 已完成或已取消的映射项不会参与刷新和顺延。手机端提供“同步完成状态到原文”，可把对应原文任务写成 `- [x]`，让规划文档的渲染结果与正式事项完成状态对齐。
5. `刷新已导入项` 支持“当前分区”或“整篇文档”范围，只刷新仍处于活动态的映射项；如果事项已被手动改动，会标记为 `CONFLICT`；如果原文行或事项找不到，会标记为 `ORPHANED`。
6. `批量顺延` 支持从某条开始到分区末尾、到文档末尾，或当前分区全部未完成项。执行后既会移动正式事项时间，也会同步改写规划文档里的时间文本。
7. 最近一次导入、刷新或顺延都可以撤销。撤销导入会删除事项并去掉 `#imported`；撤销刷新会把事项恢复到导入时内容；撤销顺延会恢复事项时间和原文时间。
8. 对于 `CONFLICT` 项，系统提供双向处理：
   - 以文档为准覆盖事项
   - 以事项为准更新原文
9. 桌面 Web 规划台不再只是“可识别、可导入”。它现在也展示当前文档标题、空状态提示、映射状态预览，以及刷新、顺延、撤销、冲突处理入口。
10. 规划文档与映射表都会进入备份/恢复快照；数据库版本升级到 `11`，通过 `MIGRATION_10_11` 创建 `planning_line_mappings`。

## 0O. 1.7.25 AI 源接入规划台识别（2026-05-15 / v1.7.25）

`1.7.25` 把前面保存的 AI 源真正用于规划台识别，同时保留本地规则兜底：

1. 设置页启用 `AI 调用配置` 且至少有一个完整启用源时，手机端和电脑端规划台顶部“识别”都会先调用 AI。
2. 内置 prompt 要求 AI 只返回 JSON，把随手写的文本拆成 todo / event / skip 候选，时间使用 `yyyy-MM-dd'T'HH:mm:ss`。
3. AI 返回会转换为现有 `PlanningParsedCandidate`，进入同一个识别预览页；用户仍可修改标题、分组、备注、DDL、日程起止时间和提醒后再导入。
4. AI 不会直接写入待办或日程。导入仍必须经过预览选择。
5. 多源按设置页顺序轮询；401/403/429/5xx、超时、DNS 失败等可重试错误会尝试下一个源。
6. AI 返回非 JSON、全部源失败、未启用或配置不完整时，会回到 `PlanningMarkdownParser` 本地规则，并在预览提示里说明。
7. 规划台识别按钮在网络识别期间显示 `识别中`，避免用户误以为点击无效；电脑端预览摘要会同步显示共享识别消息。

## 0N. 1.7.24 自然 DDL 与多 AI 源配置（2026-05-15 / v1.7.24）

`1.7.24` 继续减少用户记语法的负担，同时把后续 AI 接入需要的多源配置模型先稳定下来：

## 1. 背景

PaykiTodo 当前已经具备较强的待办、日程、提醒、全屏提醒和电脑同步能力，但这些能力主要覆盖任务管理链路的最后一环。

用户当前真实工作流是：

```text
先在纸上 / Word / Excel / 备忘录中写出接下来要做什么
  -> 再大致拆分任务
  -> 再粗略规划每一天
  -> 再对明天或今天做细化时间安排
  -> 最后才录入 PaykiTodo 的待办 / 日程 / DDL
```

因此，PaykiTodo 如果只优化最后的提醒和日程编辑，会仍然错过用户实际使用链路的前半段。

规划台要解决的问题是：

```text
让 PaykiTodo 从“最终提醒工具”前移为“从想法到执行计划”的入口。
```

## 2. 目标

### 2.1 第一阶段目标

第一阶段不做复杂拖拽、不做甘特图、不做过重的项目管理系统。

先做一个类似备忘录 / Obsidian 的自由文本规划界面：

- 能像备忘录一样自由写 Markdown。
- 支持多篇规划文档，默认打开最后一次打开的文档。
- 能识别 `- [ ]` 待办项。
- 能识别大任务下面的小任务。
- 能识别当前已实现标签，例如 `#task`、`#ddl 5.28`、`#remind 5,15`、`#group 课程`、`#schedule 10:00-11:30`。
- 能把识别出的任务转为 PaykiTodo 待办。
- 能把识别出的时间段转为 PaykiTodo 日程。
- 手机端和电脑端都能使用。

### 2.2 非目标

第一阶段暂不做：

- 拖拽任务到某一天。
- 拖拽任务到日历时间轴。
- 甘特图。
- 复杂项目依赖关系。
- 多人协作。
- 云同步。
- AI 自动规划。

这些可以作为后续功能，但不是第一版的重点。

## 3. “规划台”和“收集箱”的关系

用户提出：没有完全弄明白规划台和收集箱的区别。

建议定义如下：

### 3.1 规划台

规划台是一个大的工作区。

它像一个可编辑的 Markdown 文档，用于写：

- 接下来要做什么；
- 大任务和小任务；
- 周计划；
- 日计划；
- 临时想法；
- 可以转成待办或日程的条目。

规划台是“写、想、拆、整理”的地方。

### 3.2 收集箱

收集箱是规划台里的一个特殊区域或默认文档。

它用于快速丢进去还没整理的内容。

例如：

```markdown
# 收集箱

- [ ] 整理保研材料
- [ ] 写课程论文
- [ ] 复习音乐史
- [ ] PaykiTodo 规划台想法
```

收集箱是“先别管结构，先收进去”的地方。

### 3.3 第一阶段实现建议

第一阶段可以不强行把两者分成两个复杂模块。

可以在抽屉中新增一个入口：

```text
规划台
```

进入后默认打开一个 Markdown 文档，文档顶部可以有几个快捷分区：

```markdown
# 收集箱

# 本周计划

# 今日三件大事

# 任务拆解
```

这样用户不会被复杂模块吓退。

## 4. Markdown 任务语法设计

设计原则：

- 尽量接近日常书写习惯。
- 不强迫用户一开始写完整语法。
- `- [ ]` 是核心识别标志。
- 标签是增强信息，不是必填。
- 没写标签也可以作为普通待办候选。

## 5. 基础任务写法

### 5.1 普通任务

```markdown
- [ ] 整理保研材料
- [ ] 写课程论文
- [ ] 复习中国音乐史
```

识别结果：

- 每一行都是一个可转化任务。
- 未写 DDL 时，默认转成无 DDL 待办，或让用户在预览界面补 DDL。

### 5.2 已完成任务

```markdown
- [x] 整理成绩单
```

识别结果：

- 默认不转入新待办。
- 可以在预览中显示为“已完成，默认跳过”。

### 5.3 大任务和小任务

推荐使用 Markdown 缩进表达父子关系：

```markdown
- [ ] 保研材料准备 #ddl 5.28
  - [ ] 整理成绩单 #ddl 5.18
  - [ ] 整理获奖证明 #ddl 5.20
  - [ ] 写个人陈述初稿 #ddl 5.24
  - [ ] 联系导师 #ddl 5.26
```

识别结果：

- 第一行可以识别为项目 / 大任务。
- 缩进子项识别为子任务。
- 如果第一阶段不做复杂项目系统，可以先把子任务转为独立待办，并在备注中写入：

```text
所属大任务：保研材料准备
```

这样既保留层级信息，又不需要一开始重构数据库。

### 5.4 多层级任务

允许写，但第一阶段只识别到两层最稳：

```markdown
- [ ] 课程论文
  - [ ] 确定选题
  - [ ] 查资料
    - [ ] 找 5 篇论文
    - [ ] 整理引用
```

第一阶段处理建议：

- 一级：大任务 / 项目候选。
- 二级：待办候选。
- 三级及以下：作为上级任务备注保留。

后续如果做项目树，再完整支持多级。

## 6. 标签语法设计

### 6.1 基础标签

```markdown
#task
#project
#today
#tomorrow
#important
```

含义：

| 标签 | 含义 |
|---|---|
| `#task` | 当前已支持：明确表示这是一条任务 |
| `#project` | 当前未支持：未来才可能表示大任务 / 项目 |
| `#today` | 当前未支持：不会自动安排到今天 |
| `#tomorrow` | 当前未支持：不会自动安排到明天 |
| `#important` | 当前未支持：不会写入重要属性 |

说明：

- `- [ ]` 本身已经足够识别任务。
- `#task` 只是增强标记，适合像 Obsidian task 插件一样的习惯。
- 未支持的语义标签不会被静默吞掉，会保留在标题中，避免用户误以为已经生效。

### 6.2 DDL 标签

推荐支持：

```markdown
#ddl 5.28
#ddl 05-28
#ddl 2026-05-28
#ddl 明天
#ddl 今天 16:30
#ddl 5.28 23:59
```

默认规则：

- `5.28`、`05-28`：缺省年份，默认当前年。
- 只有日期没有时间：默认 23:59。
- `今天 16:30`：当天 16:30。
- `明天`：明天 23:59。

示例：

```markdown
- [ ] 交课程论文 #ddl 5.28
- [ ] 今晚整理材料 #ddl 今天 22:30
```

### 6.3 分组 / 目标标签

推荐支持：

```markdown
#group 保研
#group 课程
#goal 保研
#goal PaykiTodo开发
```

第一阶段可以先映射到现有“分组”。

例如：

```markdown
- [ ] 整理导师名单 #group 保研 #ddl 5.20
```

识别结果：

- 标题：整理导师名单
- 分组：保研
- DDL：2026-05-20 23:59

### 6.4 提醒标签

推荐支持：

```markdown
#remind 5
#remind 5,15
#remind 16:30
#remind 05-28 15:00
```

复用当前 PaykiTodo 已有提醒时间输入规则：

```text
5,15,16:30,2:30 pm,明天 16:30,周五 16:30,5.10 15:00,5月10日 14:30,2026-05-10 14:30
```

如果没有写 `#remind`，第一阶段建议默认：

- 待办：如果有 DDL，默认提前 5 分钟全屏提醒，可在设置中调整。
- 日程：默认开始前 5 分钟全屏提醒，可在设置中调整。

这里符合用户希望“简单写，强提醒默认补上”的诉求。

### 6.5 日程标签

推荐支持：

```markdown
#schedule 10:00-11:30
#schedule 2026-05-20 10:00-11:30
#schedule 明天 14:00-16:00
```

示例：

```markdown
- [ ] 复习音乐史 #schedule 明天 14:00-16:00
```

识别结果：

- 转为日程，而不是普通待办。
- 标题：复习音乐史。
- 时间：明天 14:00-16:00。
- 默认提醒：开始前 5 分钟全屏提醒。

## 7. 更轻量的日程自然写法

用户提到点选式添加日程不自然，更习惯写：

```text
几点到几点干什么
```

因此规划台中应额外支持“轻量日程行”。

### 7.1 推荐写法

```markdown
- [ ] 10:00-11:30 复习音乐史
- [ ] 14:00-16:00 写课程论文
- [ ] 明天 19:30-21:00 整理保研材料
- [ ] 5.20 09:00-10:30 联系导师
```

解析规则：

- 行首出现 `HH:mm-HH:mm`：识别为日程候选。
- 行首出现 `明天 HH:mm-HH:mm`：识别为明天日程。
- 行首出现 `M.d HH:mm-HH:mm`：识别为当前年的指定日期日程。
- 标题取时间段后面的文字。
- 默认提醒按设置补上，例如开始前 5 分钟全屏提醒。

### 7.2 为什么需要这个

因为它比完整批量导入语法更接近日常规划：

```markdown
# 明天

- [ ] 09:00-10:30 写论文
- [ ] 10:45-11:30 整理材料
- [ ] 14:00-16:00 复习
```

这符合用户“像在空白文档里写计划”的习惯。

## 8. 文档内日期上下文

为了减少重复写日期，可以支持标题上下文。

例如：

```markdown
# 2026-05-20

- [ ] 09:00-10:30 整理材料
- [ ] 14:00-16:00 写论文
```

或：

```markdown
# 明天

- [ ] 09:00-10:30 复习音乐史
- [ ] 14:00-16:00 写课程论文
```

解析规则：

- 如果任务行没有日期，但上方最近的标题是日期 / 今天 / 明天 / 周几，则继承该日期。
- 如果没有日期上下文，则默认今天，或在预览中要求确认。

## 9. 转化流程设计

规划台不应该用户一写就自动创建待办 / 日程。

建议流程：

```text
编辑 Markdown
  -> 点击“识别”
  -> 进入识别预览
  -> 用户勾选要导入的条目
  -> 用户选择“导入为待办 / 导入为日程 / 自动按识别结果导入”
  -> 写入 PaykiTodo 数据库
```

这样可以避免误识别造成污染。

## 10. 识别预览界面

识别结果建议按类型分组：

```text
待办候选
日程候选
项目 / 大任务候选
无法识别
已完成 / 跳过
```

每条显示：

- 原文；
- 解析出的标题；
- 类型；
- DDL / 起止时间；
- 分组；
- 提醒；
- 是否同时创建待办；
- 是否会导入。

允许用户在预览界面修正。

顶部栏：规划台                  [识别] [更多]

## 17B. AI 规划助手后续设计建议

本节记录 2026-05-14 对 AI 接入的后续展望。该能力不属于 `1.7.2`，当前版本没有接入任何 AI API。

### 17B.1 适合 AI 负责的部分

AI 不应该替代 PaykiTodo 的本地提醒和数据库逻辑。更适合它做的是“候选生成”：

1. 将自然语言规划整理成待办 / 日程候选。
2. 对大任务给出拆解建议。
3. 根据现有规划文档、待办和日程建议“每日三件大事”。
4. 发现目标偏航，例如长时间只推进软件开发而忽略保研、课程作业等高优先级任务。

### 17B.2 成本和供应商策略

GPT 系列模型直接大量调用可能成本较高。后续应设计为 OpenAI-compatible Provider，而不是写死某一家：

```text
AIProvider
  - OpenAI
  - DeepSeek
  - Qwen
  - GLM
  - OpenRouter
  - 本地 Ollama / 局域网电脑模型
```

设置项建议包括：

```text
Base URL
API Key
Model Name
请求超时
是否允许发送规划文档全文
```

### 17B.3 本地规则优先

默认流程应是：

```text
本地规则解析
  -> 识别失败或用户主动点“AI 整理”
  -> 调用 AI
  -> 生成候选
  -> 用户预览确认
  -> 本地校验
  -> 导入待办 / 日程
```

明确的 `10:00-12:30 作业1`、`#ddl 5.28`、`#remind 5,15` 这类输入应继续由本地规则处理，不应浪费 API 调用。

### 17B.4 安全边界

AI 必须受控：

1. AI 不能直接创建、修改、删除数据库里的待办或日程。
2. AI 只能输出结构化候选，例如 JSON。
3. 候选必须进入现有预览界面。
4. 用户确认后，仍要走 PaykiTodo 本地校验。
5. AI 不得绕过 DDL、提醒时间、循环规则等本地校验。
6. API Key 不应写死进 APK；个人版本可先允许用户在设置中填写，后续公开发布时应考虑后端代理或更安全的密钥方案。

### 17B.5 建议版本路线

```text
1.7.x：继续打磨规划台 Markdown/任务渲染、本地解析与 AI 候选识别
1.8.0：规划台映射闭环第一版，形成“导入 -> 跟踪 -> 刷新/顺延/撤销/冲突处理”的上游规划文档能力
1.8.1：规划台 AI 识别触发收紧，桌面导入不再隐式识别，并增加单个 AI 源测试连接
1.8.2：公告迁移到规划台多公告行，小组件改为今日看板，AI Base URL endpoint 兼容修复
1.8.3：清理旧设置公告字段，放宽公告识别，增加规划台预览公告高亮和电脑端公告横幅，优化 widget 深色模式与刷新查询
1.8.6：Android 今日看板小组件改为卡片式看板视觉，日程卡显示日期块、竖色条、标题、时间和地点
1.8.7：Android 今日看板小组件继续改向每日看板结构，增加问候卡、渐变看板底板、待办分组色条和聚合日程看板卡
1.8.9：Android 今日看板小组件叠加每日看板背景图和遮罩，头部改为圆形菜单按钮 + 标题结构，公告改为橙色横幅，卡片层级更接近日常看板
1.8.5：设置页 AI 源支持通过 `/models` 拉取可用模型并下拉选择，保留手动模型名兜底
1.8.4：电脑端公告横幅改为长度感知滚动并适配深色模式，小组件公告/待办/日程行支持深链入口
1.8.x：每日三件大事、偏航检测、周计划建议，以及更自动化的规划同步体验
```

## 17A. 本地解析和自然输入补充规范

本节补充第一轮用户反馈后的解析规则。第一阶段必须本地完成，不依赖 AI 模型。

### 17A.1 本地解析原则

规划台解析器应作为本地规则引擎实现。

优先级建议：

```text
显式标签 #schedule / #ddl / #remind
  > 行首自然日程
  > Markdown checkbox
  > 日期上下文标题
  > 默认导入策略
```

这样可以保证用户写了明确标签时尊重标签；没写标签时，也能识别自然写法。

### 17A.2 需要覆盖的日期格式

| 类型 | 示例 |
|---|---|
| 月日点号 | `5.28`、`05.28` |
| 月日短横线 | `5-28`、`05-28` |
| 中文月日 | `5月28日`、`05月28日` |
| 完整日期短横线 | `2026-05-28` |
| 完整日期点号 | `2026.05.28` |
| 完整中文日期 | `2026年5月28日` |
| 相对日期 | `今天`、`明天`、`后天` |

默认规则：

- 未写年份时，默认当前年。
- 只有日期没有时间时，默认 23:59。
- 如果解析出的 DDL 早于当前时间，识别预览中标红提示，不静默改到明年。

### 17A.3 需要覆盖的时间格式

| 类型 | 示例 |
|---|---|
| 24 小时英文冒号 | `14:30` |
| 24 小时中文冒号 | `14：30` |
| 12 小时制 | `2:30 pm`、`2:30pm`、`11:55PM` |
| 单数字小时 | `9:00`、`2:30 pm` |

### 17A.4 日期和时间组合

日期和时间之间允许空格或逗号：

```text
5.28 14:30
5.28,14:30
5月28日 11:55pm
2026-05-28 14:30
2026年5月28日, 2:30 pm
```

### 17A.5 轻量日程自然写法

第一阶段必须支持用户最自然的写法：

```text
10:00-12:30 作业1
10:00~12:30 作业1
10:00 至 12:30 作业1
10:00到12:30 作业1
明天 19:30-21:00 整理保研材料
5.28 14:30-16:00 写论文
5月28日 11:55pm-次日00:30 整理材料
```

解析规则：

- 行首是时间段时，默认识别为日程。
- 如果行内没有日期，则读取最近的日期上下文标题。
- 如果没有日期上下文，默认策略待进一步确认：默认今天，或在预览中要求确认。
- 标题取时间段后面的文字。
- 导入时默认同时创建一条待办，待办 DDL 等于日程结束时间。
- 识别预览中应允许取消“同时创建待办”。

### 17A.6 日期上下文标题

规划台应支持通过标题提供日期上下文：

```markdown
# 明天

- [ ] 09:00-10:30 写论文
- [ ] 14:00-16:00 复习
```

标题可识别：

```text
# 今天
# 明天
# 后天
# 周五
# 5.28
# 5月28日
# 2026-05-28
```

## 17B. 手机端输入辅助设计

标签和 Markdown 语法属于高级能力，不能让用户在手机中文输入环境里手动找符号。

### 17B.1 输入法上方快捷栏

手机端规划台在输入法弹出时，应显示一条横向快捷栏：

```text
[任务] [子任务] [缩进] [减少缩进] [DDL] [日程] [提醒] [分组] [今天] [明天]
```

行为：

- 点 `任务`：插入 `- [ ] `。
- 点 `子任务`：插入 `  - [ ] `，或把当前行改成子任务。
- 点 `缩进`：当前行增加两个空格缩进。
- 点 `减少缩进`：当前行减少一级缩进。
- 点 `DDL`：插入 `#ddl `。
- 点 `日程`：插入 `#schedule `。
- 点 `提醒`：插入 `#remind `。
- 点 `分组`：插入 `#group `。
- 点 `今天` / `明天`：插入对应日期上下文或标签。
- 长按按钮：显示该按钮用途和例子。

### 17B.2 回车续写规则

为了解决“大任务下面输入小任务”的问题，编辑器应支持列表续写：

- 当前行是 `- [ ] 大任务`，按回车后自动插入下一行 `- [ ] `。
- 点击 `子任务` 后，下一行变成 `  - [ ] `。
- 当前行是 `  - [ ] 子任务`，按回车后继续生成同级子任务 `  - [ ] `。
- 如果当前行是空的子任务行，再按一次回车，则退出缩进，回到 `- [ ] `。

这套行为让用户不必理解 Markdown 缩进，也能自然写出父任务和子任务。

### 17B.3 手机端不强迫完整 Markdown

用户可以直接写：

```text
10:00-12:30 作业1
整理保研材料 5.28
明天 14:00-16:00 复习音乐史
```

识别器应尽量识别这些自然行。`- [ ]` 和标签只是增强准确度，不是唯一入口。
