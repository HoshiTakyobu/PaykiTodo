# 第二部分：日程打卡 / 签到（时间追踪）

> 这是 1.11.0 的核心新功能。原 1.11.0 prompt 的 C1–C6 整体保留，C7（与专注的关系）随专注模式一同移除。

## 功能定位

用户希望在日程进行期间记录**实际投入时间**：

- 绑定到**日程**（不是待办）
- **正计时**（签到开始，签退停止），不是倒计时
- 支持一次日程中**多次签到/签退**（中途可暂停）
- 日程结束时呈现**总投入时间和投入率**
- **可选功能**，需要在日程编辑器里开启

### 用户场景

```
日程：14:00-16:00 写课程论文（已开启打卡）

14:03 签到 → 开始计时
14:45 签退 → 暂停
14:52 签到 → 继续
15:48 签退 → 暂停
15:55 签到 → 继续
16:05 完成日程

结果：总投入 1h53m / 计划 2h / 投入率 94% / 3 次签到
```

## V4. 识图 Prompt

system prompt（固定）：
```
你是一个课表/日程识图助手。用户给你一张课表或日程截图，请你识别其中所有的课程或日程，按以下格式逐行输出 Markdown，不要任何解释，不要 code fence：

H:MM-HH:MM, 课程名, @地点
H:MM-HH:MM, 课程名, @地点

规则：
1. 一行一个日程
2. 时间段用 24 小时制，连字符分隔
3. 地点前加 @，地点不存在就省略 @ 段

5. 如果识别到的日程跨多个工作日（例如每周一三五），在每行末尾加 "（每周X）" 或 "（周一/三/五）" 等中文标注
```

## V5. 识别流程

1. 用户选图 → 显示进度对话框（不可取消，含"AI 识别中…可能需要 10-30 秒"）
2. AI 调用完成后，把返回的 Markdown **追加到当前规划文档末尾**（在新建段落分隔的空行后）
3. 自动滚动到追加位置，提示"已追加 N 条识别结果，请检查后用「识别」按钮进入预览导入"
4. 用户像平时一样用现有 `识别` 按钮触发 local + AI 双轨解析，进入现有 preview 流程

**不直接进入 import preview**：让用户先肉眼检查识别结果（AI 识图有错率），手动微调 Markdown 后再正式 parse。

## V6. 错误处理

- 网络失败 / 超时 → 弹 toast "图片识别失败：<原因>"，不污染规划文档
- AI 返回空内容 → 提示"未能从图中识别出日程"
- 图片过大无法压缩 → 提示"图片过大，请裁剪后重试"

## M3. 多选交集筛选

> ⚠️ **关键设计决策**：用户要求"选了 A 和 B 只显示同时属于 A 和 B 的待办"（交集），但当前数据模型每个 todo 只能挂一个 `groupId`。要实现真正的交集筛选，需要把分组关系从 "一对多" 改为 "多对多"。

### 方案：保留 `groupId` 作为主分组 + 新增多对多关联表

#### 新增 entity
`app/src/main/java/com/example/todoalarm/data/TodoGroupTag.kt`

```kotlin
@Entity(
    tableName = "todo_group_tags",
    primaryKeys = ["todoId", "groupId"],
    indices = [
        Index(value = ["todoId"], name = "index_todo_group_tags_todoId"),
        Index(value = ["groupId"], name = "index_todo_group_tags_groupId")
    ]
)
data class TodoGroupTag(
    val todoId: Long,
    val groupId: Long
)
```

#### 数据库迁移（合并到 `MIGRATION_17_18`）
```sql
CREATE TABLE IF NOT EXISTS todo_group_tags (
    todoId INTEGER NOT NULL,
    groupId INTEGER NOT NULL,
    PRIMARY KEY (todoId, groupId)
);
CREATE INDEX IF NOT EXISTS index_todo_group_tags_todoId ON todo_group_tags(todoId);
CREATE INDEX IF NOT EXISTS index_todo_group_tags_groupId ON todo_group_tags(groupId);

-- 把现有 todo 的 groupId 反向插入 join 表，保持升级后筛选行为不变
INSERT OR IGNORE INTO todo_group_tags (todoId, groupId)
SELECT id, groupId FROM todo_items WHERE groupId != 0 AND itemType = 'TODO';
```

#### DAO 新增

```kotlin
@Insert(onConflict = OnConflictStrategy.REPLACE)
suspend fun insertTodoGroupTags(tags: List<TodoGroupTag>)

@Query("DELETE FROM todo_group_tags WHERE todoId = :todoId")
suspend fun clearTodoGroupTags(todoId: Long)

@Query("SELECT groupId FROM todo_group_tags WHERE todoId = :todoId")
suspend fun getGroupIdsForTodo(todoId: Long): List<Long>

@Query("DELETE FROM todo_group_tags WHERE groupId = :groupId")
suspend fun clearTagsByGroup(groupId: Long)

/**
 * 多分组交集查询：返回同时属于所有 selectedGroupIds 的活跃 todo。
 * 通过 GROUP BY + HAVING COUNT 实现集合的 ALL 关系。
 */
@RawQuery(observedEntities = [TodoItem::class, TodoGroupTag::class])
fun observeActiveTodosByGroupIntersection(query: SupportSQLiteQuery): Flow<List<TodoItem>>
```

Repository / ViewModel 中根据当前选中的 `Set<Long>` 动态拼装 SQL：
- 空集合 → 用现有 `observeActiveTodoItems(null)`
- 1 个 → 用现有 `observeActiveTodoItems(groupId)`
- 2+ → 拼 `SELECT t.* FROM todo_items t JOIN todo_group_tags g ON t.id = g.todoId WHERE t.completed=0 AND t.canceled=0 AND t.itemType='TODO' AND g.groupId IN (?, ?, ...) GROUP BY t.id HAVING COUNT(DISTINCT g.groupId) = N ORDER BY t.dueAtMillis ASC`

## P8. `DesktopSyncCoordinator` 消除 `runBlocking`

### 关键修正

原 1.11.0 prompt 的方案"直接去掉 `runBlocking` 改为 `suspend` 调用"会编译失败 —— `DesktopSyncServer` 的 `requestHandler` 签名是同步函数类型，由 `clientExecutor` 线程池同步调用。

### 正确改法

把 server 的 handler 类型改为 suspend：

`DesktopSyncServer.kt`：
```kotlin
class DesktopSyncServer(
    private val port: Int,
    private val requestHandler: suspend (method: String, path: String, body: String, headers: Map<String, String>) -> Response
)

private fun handleClient(socket: Socket) {
    socket.use { client ->
        // ...
        val response = runBlocking { requestHandler(method, path, body, headers) }
        // ...
    }
}
```

`DesktopSyncCoordinator.kt`：
- `handleRequest` 改为 `suspend fun`
- 内部所有 `runBlocking { ... }` 改为直接 `await` / 直接 suspend 调用
- 这样 `runBlocking` 只在网络客户端线程上出现一次，且这条线程本来就是要等响应的，不会再阻塞业务 IO 线程池
