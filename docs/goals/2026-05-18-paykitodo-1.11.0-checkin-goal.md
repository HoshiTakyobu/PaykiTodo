## P5. `DesktopSyncCoordinator` 消除 `runBlocking`

### 当前问题
54 处 `runBlocking` 在 IO 线程池上阻塞。多个浏览器同时请求时线程池被占满。

### 改动策略
`DesktopSyncCoordinator` 的 handler 方法已经在 `scope.launch(Dispatchers.IO)` 内部。直接去掉 `runBlocking` 包裹，改为 `suspend` 调用。如果 handler 签名不是 suspend，把整个 handler body 包在 `scope.launch` 里（检查现有代码是否已经这样做了，如果是则只需去掉内部的 `runBlocking`）。

# 第二部分：日程打卡/签到（时间追踪）

## 功能定位

用户希望在日程进行期间记录**实际投入时间**。和"专注模式"（绑定待办的番茄钟）不同：

- 绑定到**日程**（不是待办）
- **正计时**（签到开始，签退停止），不是倒计时
- 支持一次日程中**多次签到/签退**（中间可能去上厕所）
- 日程结束时呈现**总投入时间和投入率**
- **可选功能**，需要在日程编辑器里开启（类似倒数日的 `countdownEnabled`）

### 用户场景

```
日程：14:00-16:00 写课程论文（已开启打卡）

14:03 签到 → 开始计时
14:45 签退 → 暂停（去上厕所）
14:52 签到 → 继续
15:48 签退 → 暂停
15:55 签到 → 继续
16:05 完成日程

结果：总投入 1h53m / 计划 2h / 投入率 94% / 3 次签到
```

## C1. 数据模型

### 新建文件
`app/src/main/java/com/example/todoalarm/data/EventCheckIn.kt`

```kotlin
@Entity(tableName = "event_check_ins")
data class EventCheckIn(
    @PrimaryKey(autoGenerate = true) val id: Long = 0,
    val eventId: Long,
    val checkInAtMillis: Long,
    val checkOutAtMillis: Long? = null,
    val durationMinutes: Int = 0
)
```

### 日程表扩展
`todo_items` 新增：
- `checkInEnabled INTEGER NOT NULL DEFAULT 0`
- `totalCheckInMinutes INTEGER NOT NULL DEFAULT 0`

### 数据库迁移
版本 +1，创建 `event_check_ins` 表 + 索引 `index_event_check_ins_eventId`，ALTER `todo_items` 加两个字段。

### DAO
```kotlin
@Insert suspend fun insertCheckIn(checkIn: EventCheckIn): Long
@Query("UPDATE event_check_ins SET checkOutAtMillis = :checkOutAt, durationMinutes = :duration WHERE id = :id")
suspend fun checkOut(id: Long, checkOutAt: Long, duration: Int)
@Query("SELECT * FROM event_check_ins WHERE eventId = :eventId ORDER BY checkInAtMillis ASC")
suspend fun getCheckInsForEvent(eventId: Long): List<EventCheckIn>
@Query("SELECT * FROM event_check_ins WHERE eventId = :eventId AND checkOutAtMillis IS NULL LIMIT 1")
suspend fun getActiveCheckIn(eventId: Long): EventCheckIn?
@Query("UPDATE todo_items SET totalCheckInMinutes = :minutes WHERE id = :id")
suspend fun updateTotalCheckInMinutes(id: Long, minutes: Int)
```

## C7. 与专注模式的关系

两者不冲突：
- 专注 = 绑定待办的倒计时番茄钟
- 打卡 = 绑定日程的正计时投入追踪
- 可以同时进行（日程签到中 + 番茄钟倒计时）
- 数据分别存 `focus_sessions` 和 `event_check_ins`

AI 日报中同时展示：
- 今日专注：X 分钟（focus_sessions）
- 今日日程投入：Y 分钟（event_check_ins）

# 第三部分：体验增强 Ideas

## I1. 快速添加（通知栏常驻快捷方式）

Android 通知栏放一个常驻低优先级通知，点击弹出"快速添加待办"输入框。不用打开 App 就能记一件事。

实现：
- 新建 `QuickAddNotificationService`，在设置里开关控制
- 通知用 `NotificationCompat.Builder` + `addAction` 或 `setContentIntent` 打开一个透明 Activity
- 透明 Activity 只有一个 `AlertDialog`：标题输入 + 可选 DDL + 确认
- 确认后写入数据库，dismiss Activity

---

## I2. 数据统计面板

抽屉新增"统计"入口，展示：
- 本周/本月完成率（完成数 / 总数）
- 各分组投入时间饼图（来自专注 + 打卡）
- 连续完成天数（streak）
- 专注时长趋势（最近 7 天柱状图）

实现：
- 新建 `StatisticsPanel.kt`
- 用 Compose `Canvas` 画简单柱状图和饼图，不引入第三方图表库
- 数据从 `focus_sessions` + `event_check_ins` + `todo_items` 聚合

---

## I3. 课程表从 ICS 导入

从教务系统导出的 `.ics` 文件直接导入为循环日程。

实现：
- 设置 → 日历 → "从 ICS 文件导入"
- 用 `DocumentFile` 选择 `.ics` 文件
- 解析 `VEVENT` 组件（标题、开始/结束时间、RRULE、地点）
- 转为 `CalendarEventDraft` 列表，进入预览确认页
- 确认后批量创建日程

## I4. 待办模板（快速创建常用任务）

用户经常创建相同结构的待办（比如"复习 XX 第 N 章"）。

实现：
