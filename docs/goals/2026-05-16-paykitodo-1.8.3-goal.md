3. 公告 text 中如果包含其他类似 `#group xxx` 的尾部 hashtag，也剥掉

## Part 6：公告排序按近期优先

### 问题

`PlanningAnnouncementParser.parse()` 第 48 行：

```kotlin
compareBy<PlanningAnnouncement> { it.startDate ?: LocalDate.MIN }
```

长期公告（`startDate=null`）会被排到最前面。但用户预期近期生效的优先，长期公告兜底。

### 任务

排序规则改为：
1. 优先显示有日期范围且包含今天的（近期生效）
2. 同样有日期范围时，按 startDate 降序（越近的越靠前）
3. 长期公告（无 startDate）排在最后
4. 同优先级内按 sourceNoteId 和 lineNumber

实现：

```kotlin
.sortedWith(
    compareBy<PlanningAnnouncement> { it.startDate == null }  // 长期排后
        .thenByDescending { it.startDate ?: LocalDate.MIN }    // 近期优先
        .thenBy { it.sourceNoteId }
        .thenBy { it.lineNumber }
)
```

### 验证

文档里同时有 `#公告 5.16 A`、`#公告 6.1 B`、`#公告 长期 C` 三条。今天看板上看到顺序：B → A → C。

## Part 7：Widget 数据加载窄查询

### 问题

`TodoWidgetService.kt:36`：

```kotlin
val items = app.repository.getAllTodos()
val notes = app.repository.getAllPlanningNotes()
```

`getAllTodos()` 把所有历史待办全拉出来再交给 `DailyBoardSnapshotBuilder` 过滤。用户用半年后 todos 上千条，每次 widget 刷新（包括每个 todo 完成时）都全表扫描。

### 任务

1. `TodoDao` 新增窄查询方法：
   ```kotlin
   @Query("SELECT * FROM todo_items WHERE completed = 0 AND canceled = 0 AND (dueAtMillis = 0 OR dueAtMillis BETWEEN :startMillis AND :endMillis) ORDER BY dueAtMillis ASC")
   fun getActiveTodosForRange(startMillis: Long, endMillis: Long): List<TodoItem>
   ```

2. `TodoRepository` 新增对应方法：
   ```kotlin
   suspend fun getActiveTodosForBoardRange(): List<TodoItem> {
       val zone = ZoneId.systemDefault()
       val today = LocalDate.now()
       val start = today.atStartOfDay(zone).toInstant().toEpochMilli()
       val end = today.plusDays(2).atStartOfDay(zone).toInstant().toEpochMilli()
       return dao.getActiveTodosForRange(start, end)
   }
   ```
   
   注意：还要包含"无 DDL 的待办"（dueAtMillis=0），所以查询条件是 `dueAtMillis = 0 OR dueAtMillis BETWEEN ...`。**但 widget 看板只关心今日和明日**，无 DDL 的不显示，所以查询只用 BETWEEN 即可，不要 `OR dueAtMillis = 0`

3. `TodoWidgetService.onDataSetChanged` 改用新方法

4. 同时检查 `DailyBoardSnapshotBuilder` 对输入的预期：是否依赖 `items` 包含全部 todos（用于推算 missed 状态）。如果是，widget 端的窄查询需要扩大范围把"已 missed 但还未完成"的旧 todo 也包含进来。建议把窗口扩大到「30 天前到 2 天后」覆盖 missed

### 验证

## Part 8：移除 Widget 双重刷新

### 问题

`TodoWidgetProvider.kt:22-27`：

```kotlin
override fun onReceive(context: Context, intent: Intent) {
    super.onReceive(context, intent)
    if (intent.action == AppWidgetManager.ACTION_APPWIDGET_UPDATE) {
        notifyWidgetDataChanged(context)
    }
}
```

`super.onReceive` 已经会路由 `ACTION_APPWIDGET_UPDATE` 到 `onUpdate` 进而 `updateWidget`。这里再调 `notifyWidgetDataChanged` 是冗余，导致每次系统刷新触发两次 `updateWidget`。

### 任务

直接删掉这个 `onReceive` 重写。`AppWidgetProvider` 默认的 `onReceive` 会正确路由所有 widget action。

### 验证

桌面 widget 行为不变，但每次系统广播只触发一次 `updateWidget`。
